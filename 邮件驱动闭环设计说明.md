# 研发补贴台账：邮件驱动闭环设计说明

## 目标

下一版的目标不是让操作人员多填表，而是把会计每月已有的账表变成系统入口。

最简单的使用方式：

1. 会计每月做完账。
2. 会计把明细账 Excel 或 CSV 发到指定邮箱。
3. 系统读取邮件附件，自动识别研发费用。
4. 申报负责人只审核不确定的项目归属。
5. 系统自动生成管理层简报。
6. CEO/CFO 收到一份很短的预警页面。

## 管理层页面原则

CEO/CFO 页面只说明问题，不展示后台工作量。

只保留：

- 预警等级：红色、橙色、绿色。
- 研发投入达标率。
- 研发投入缺口。
- 补贴到账率。
- 本月最需要知道的三件事。

不展示：

- 单笔费用明细。
- 凭证和发票。
- 项目字段维护。
- 费用拆分过程。
- 邮件解析日志。

## 后台处理内容

这些内容放在后台，不给管理层页面增加复杂度：

- 邮件收取。
- 附件保存。
- Excel/CSV 解析。
- 研发费用科目识别。
- 杭州微新和深圳微智主体识别。
- 项目匹配。
- 人员费用分摊。
- 不确定费用的人工审核。
- 月报 HTML 生成。
- 邮件或企业微信发送。

## 每月流程

1. 会计发送邮件。

   建议邮件标题：

   `研发费用月度明细-2026-06-杭州微新`

   或：

   `研发费用月度明细-2026-06-深圳微智`

2. 邮件附件上传。

   附件建议包含这些列：

   - 日期
   - 法人主体
   - 会计科目
   - 摘要
   - 金额
   - 凭证号
   - 部门或项目

3. 系统自动归集。

   系统先按规则判断：

   - 是否属于研发费用。
   - 属于杭州微新还是深圳微智。
   - 属于哪个补贴项目。
   - 是否需要人员费用分摊。

4. 申报负责人审核。

   系统只把不确定的记录列出来给负责人处理。

   例如：

   - 摘要不清楚。
   - 科目不是标准研发费用。
   - 同一人员费用可能要分到多个项目。
   - 杭州和深圳主体不明确。

5. 系统生成简报。

   简报重点只回答：

   - 哪个主体研发投入不够。
   - 差多少钱。
   - 哪个项目风险最高。
   - 本月需要谁处理什么。

6. 自动发送。

   默认发给管理层邮箱，也可以后续接企业微信群机器人。

## 需要准备的账号

建议单独建一个公司邮箱，例如：

`subsidy-ledger@synbiome.cn`

这个邮箱只用于收会计账表和发送月报。

不要把个人邮箱密码写进网页文件，也不要放进 GitHub。

正式实现时，邮箱授权密码应放在腾讯云后台的环境变量里。

## 下一版最小功能边界

第一版自动化只做这些：

- 读取一个固定邮箱里的新邮件。
- 下载 Excel/CSV 附件。
- 识别基础字段。
- 生成导入批次。
- 把不确定记录放入待审核列表。
- 根据审核后的数据更新台账。
- 生成管理层 HTML 简报。

暂时不做：

- 直接连接完整会计系统。
- 自动读取所有凭证附件。
- 自动判断复杂税务或审计口径。
- 多级审批流。
- 手机小程序。

## 验收标准

下一版做好后，用一封测试邮件验证：

1. 会计发一封带明细账附件的邮件。
2. 系统能看到导入批次。
3. 明确费用能自动进入对应项目。
4. 不确定费用进入待审核列表。
5. 审核完成后，平衡总览数字更新。
6. 系统生成一份管理层 HTML 简报。
7. 简报可以发到指定邮箱。

## 当前结论

当前网页继续保留手动录入和管理层简报。

邮件自动导入属于下一版后台能力。它的价值是减少人工填表，让会计只发送账表，系统负责识别、提醒和汇报。
