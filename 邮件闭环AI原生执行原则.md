# 邮件闭环 AI 原生执行原则

## 结论

这三条不是建议，而是项目执行硬约束。

后续判断一个方案是否合格，只看一件事：

**它有没有把人的注意力从重复搬运和重复确认里解放出来。**

## 1. 禁止前端确认

前端页面不能成为每月流程里的固定确认入口。

允许前端做三件事：

- 展示结果。
- 展示预警。
- 展示系统无法判断的异常项。

不允许前端做这些事：

- 每月让负责人点一次“确认导入”。
- 每月让会计确认一遍“数据无误”。
- 每月让管理层登录系统检查是否有问题。
- 把系统已经能判断的内容再交给人确认。

例外情况：

只有系统确实判断不了时，才进入人工处理。

例如：

- 费用归属杭州还是深圳不清楚。
- 人员费用分摊规则缺失。
- 会计科目不标准但摘要疑似研发。
- 政策口径需要财务或申报负责人判断。

如果同一类异常每个月重复出现，就不能一直靠前端确认解决，要把规则补进后台。

## 2. 禁止任何人工搬运

系统目标不是把 Excel 从一个地方搬到另一个地方。

不允许形成长期固定动作：

- 从会计系统下载文件。
- 再上传到台账系统。
- 从邮件里下载附件。
- 再复制粘贴到网页。
- 把表格数字手动搬进月报。

允许的临时动作：

- 初始项目资料一次性导入。
- 第一次测试时人工发送样表。
- 接口未打通前的短期兜底。

但所有临时动作都要有退出条件：

- 谁负责。
- 用到什么时候。
- 后续用哪个接口替代。

## 3. 找到接口方案

每一个重复动作，都先找接口方案。

优先级如下：

1. 会计系统官方 API 或开放接口。
2. 会计系统定时导出到指定邮箱。
3. 公司邮箱通过 IMAP/SMTP 自动收发。
4. 云盘或对象存储的自动文件投递。
5. 手工上传 CSV 或 Excel。

第 5 项只能作为临时兜底，不能作为正式长期流程。

## 本项目执行口径

前端不是流程入口，前端只是驾驶舱和异常队列。

管理层不应该主动登录系统找问题，系统应该每月把问题推给管理层。

会计不应该每月重复填写台账，系统应该从会计系统、邮箱或接口里自动取得明细。

项目负责人不应该审核所有记录，只处理系统无法判断的少数异常。

如果一个流程需要某个人每月做同样的点击、下载、上传、复制、粘贴，就视为设计未完成。

## 对当前 MVP 的影响

当前邮件方案保留，但定位调整为：

**邮箱是过渡接口，不是长期人工搬运流程。**

第一阶段可以用邮件附件跑通闭环：

- 自动收邮件。
- 自动读附件。
- 自动归集研发费用。
- 自动生成预警月报。

但正式执行时，要同步确认会计系统是否支持：

- 固定报表订阅。
- 定时发送邮件。
- API 拉取明细账。
- 自动导出到云盘或对象存储。

如果会计系统可以直接接口获取明细账，就优先走接口，不再要求会计每月人工发附件。

## 验收标准

下一版不能只验收“页面能用”。

必须同时验收：

- 是否没有固定的前端确认步骤。
- 是否没有固定的人工下载上传步骤。
- 是否已经查过会计系统接口或自动导出能力。
- 是否只有异常项才需要人处理。
- 是否能自动生成并推送管理层月报。

只要仍然依赖人每月搬运数据，就不能称为自动化完成。
