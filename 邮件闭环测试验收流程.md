# 邮件闭环测试验收流程

## 用途

这份流程用于正式开发完成后的验收。

只有按这里跑通，才可以说邮件闭环开始具备自动运行能力。

## 测试前准备

测试前必须准备：

1. 专用邮箱。
2. 邮箱授权密码已经配置到腾讯云环境变量。
3. 一份真实导出的 Excel 或 CSV 研发费用明细账。
4. 已完成 `附件字段映射确认表.csv`。
5. 已确认月报收件人。
6. 已确认 `管理层月报样张.html` 的简报结构。
7. 已确认会计系统 API、定时导出、邮箱自动投递或云盘投递的可行性结论。
8. 已确认测试邮件只是验收方式，不代表长期人工搬运流程。

## 测试邮件

由会计发送测试邮件到专用邮箱。

这一步只用于验收系统是否能读邮件和附件。正式流程应优先使用接口、定时邮件或自动导出。

邮件标题建议：

`研发费用月度明细-2026-06-杭州微新`

或：

`研发费用月度明细-2026-06-深圳微智`

附件使用真实导出的 Excel 或 CSV。

## 验收步骤

### 第一步：系统读取邮件

验收标准：

- 系统能找到测试邮件。
- 系统能识别邮件标题里的月份。
- 系统能识别邮件标题或附件里的法人主体。

失败处理：

- 如果找不到邮件，先检查邮箱授权和收件服务器配置。
- 如果月份或主体识别失败，先检查邮件标题和附件字段。

### 第二步：系统解析附件

验收标准：

- 系统能读取 Excel 或 CSV。
- 系统能识别日期、主体、科目、摘要、金额。
- 系统能生成一批导入记录。

失败处理：

- 如果列名不一致，更新 `附件字段映射确认表.csv`。
- 如果金额列无法识别，让会计重新导出样表。

### 第三步：系统判断费用

验收标准：

- 明确的研发费用进入候选研发费用。
- 明确非研发费用被忽略。
- 人员工资、社保、公积金等进入待审核。
- 主体或项目不明确的记录进入待审核。

失败处理：

- 如果明显研发费用没有识别，补充科目或摘要规则。
- 如果明显非研发费用被导入，补充忽略规则。

### 第四步：负责人审核不确定记录

验收标准：

- 张英能看到待审核记录。
- 张英能确认项目归属。
- 张英能确认人员费用分摊。
- 审核后记录能进入台账。
- 系统能自动判断的记录不需要张英逐条确认。

失败处理：

- 如果审核人看不到记录，检查权限。
- 如果分摊比例无法录入，先用人工备注方式记录，再进入下一版优化。

### 第五步：台账数字更新

验收标准：

- 杭州微新和深圳微智的研发投入金额更新。
- 项目研发投入金额更新。
- 研发投入缺口重新计算。
- 补贴到账率不被错误改动。

失败处理：

- 如果金额不一致，导出导入批次和原始附件逐行核对。

### 第六步：生成管理层 HTML 月报

验收标准：

- 月报显示预警等级。
- 月报显示研发投入达标率。
- 月报显示研发投入缺口。
- 月报显示补贴到账率。
- 月报显示本月最重要三件事。
- 月报不显示单笔费用明细。

失败处理：

- 如果信息太多，删减为管理层只看结论。
- 如果缺少关键数字，检查台账汇总逻辑。

### 第七步：发送月报

验收标准：

- 系统能发出 HTML 月报。
- 指定收件人能收到。
- 邮件正文或附件不包含邮箱授权密码。
- 邮件正文或附件不包含技术错误日志。

失败处理：

- 如果无法发送，检查发件服务器和授权密码。
- 如果收不到，检查收件人列表和垃圾邮件。

### 第八步：检查是否仍有人在搬运

验收标准：

- 没有每月固定前端确认步骤。
- 没有每月固定人工上传步骤。
- 没有复制粘贴数字生成月报。
- 临时人工发送测试邮件已有退出条件。

失败处理：

- 如果还需要人每月重复搬运，回到接口或自动导出方案重新设计。

## 验收结果判断

全部八步通过：

**可以进入试运行。**

任意一步失败：

**不能上线，只能继续测试。**

## 测试留证

每次测试建议保存：

- 测试邮件发送时间。
- 测试附件文件名。
- 导入批次编号。
- 待审核记录数量。
- 审核人。
- 台账更新前后金额。
- HTML 月报截图或文件。
- 月报发送成功记录。
- 接口或自动导出能力结论。
- 临时人工兜底的退出条件。
