# 邮件闭环实施计划

## 一句话

系统以后自动取得会计账表，负责导入、提醒和生成管理层简报。邮件附件可以作为第一阶段过渡接口，但不能长期靠人每月搬运。

## 为什么这样做

现在最怕的问题是：杭州微新和深圳微智的研发费用分配不准，导致某一边达不到政府要求，最后补贴拿不到。

所以下一版不追求复杂审批系统，只解决一个关键问题：

**把会计账表里的研发费用及时导进台账，系统每月自动提醒哪边缺口大。**

## 第一步：准备专用邮箱

建议准备一个公司邮箱，例如：

`subsidy-ledger@synbiome.cn`

这个邮箱只做两件事：

- 接收会计发来的研发费用明细账。
- 发出每月管理层简报。

不建议直接用个人邮箱长期发送月报，因为以后人员变化或密码变化会影响系统。

## 第二步：确认数据怎么自动进来

优先确认会计系统是否支持：

- API 拉取明细账。
- 报表订阅。
- 定时发送邮件。
- 自动导出到云盘或对象存储。

如果暂时都不支持，测试阶段可以由会计导出研发费用明细账，发到专用邮箱。

邮件标题建议固定：

`研发费用月度明细-2026-06-杭州微新`

或：

`研发费用月度明细-2026-06-深圳微智`

附件可以是 Excel 或 CSV。

字段可以参考：

- `会计研发费用明细样表.csv`
- `会计样表填写说明.md`

测试邮件可以直接套用：

- `测试邮件模板.md`

这只是测试或临时兜底，不作为长期每月固定动作。

## 第三步：系统后台处理

系统后台自动做这些事情：

1. 收邮件。
2. 下载附件。
3. 识别会计科目里属于研发费用的记录。
4. 判断属于杭州微新还是深圳微智。
5. 能判断项目的，自动归集。
6. 判断不准的，放到待审核列表。

系统能判断的记录不再让人到前端重复确认。

## 第四步：负责人只审核不确定项

申报负责人不需要看所有明细，只处理系统不确定的记录。

常见不确定项：

- 人员工资需要分摊到多个深圳课题。
- 摘要里没有写清楚项目。
- 科目不是标准研发费用，但可能和研发相关。
- 主体是杭州还是深圳不明确。

## 第五步：管理层只收简报

CEO/CFO 不看明细表，只收一份 HTML 简报。

简报只回答四个问题：

- 是否预警。
- 研发投入还差多少。
- 补贴到账比例是多少。
- 本月最需要处理哪三件事。

## 需要注意

邮箱授权密码不能写在网页文件里，也不能放 GitHub。

正式实现时，应放在腾讯云后台环境变量里。

前端页面只显示结果、预警和异常项，不做每月固定确认。

人工下载、上传、复制、粘贴只能作为测试或临时兜底。

## 暂时不做

- 不做完整会计系统深度直连，但要先查接口或自动导出能力。
- 不上传所有凭证和发票。
- 不让管理层看明细。
- 不做复杂审批流。
- 不做手机小程序。

## 验收方法

下一版开发完成后，用一封测试邮件验收：

1. 会计发送 Excel 或 CSV。
2. 系统出现一个导入批次。
3. 明确的研发费用自动进入台账。
4. 不确定的费用进入待审核。
5. 负责人审核后，平衡总览数字更新。
6. CEO/CFO 收到 HTML 简报。
7. 没有固定前端确认步骤。
8. 没有固定人工搬运步骤。
9. 已经记录会计系统接口或自动导出能力的判断结论。

## 开发前先确认

正式开发前，先看 `邮件闭环设计确认单.md`。

确认单只问几个关键问题：

- 用哪个专用邮箱。
- 月报发给谁。
- 每月哪一天发送。
- 会计附件用 Excel 还是 CSV。
- 不确定费用由谁审核。
- 管理层是否只看简报。
- 会计系统有没有 API、定时导出或自动投递能力。
- 人工发送邮件是否只是测试或临时兜底。
